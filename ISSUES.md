### #1 회원 관리 페이지를 실제 API에 연결

**난이도:** 낮음 · **영역:** 프론트

`/admin/members`가 데모 데이터(`lib/admin.ts`의 `members()`)를 씁니다.

**이미 있는 것 (백엔드):**
- `GET /admin/members?q=` — 검색어로 회원 조회
- `PATCH /admin/members/:id/suspend` — 정지/해제 (body: `{ suspended: boolean }`)

**할 일:**
1. `coliving/src/lib/api/admin.ts`에 `listMembers()`, `suspendMember()` 추가
2. `app/admin/members/page.tsx`를 실 API로 교체
3. 정지 버튼이 실제로 DB에 반영되는지 확인

**주의:** 정지는 되돌리기 어려운 동작입니다. `admin/approvals/page.tsx`의 "거부" 버튼처럼
2단계 확인(한 번 더 클릭)을 넣어주세요.

**완료 기준:** 회원 정지 → 새로고침 → 정지 상태가 유지됨

### #2 신고 관리 페이지를 실제 API에 연결

**난이도:** 낮음~중 · **영역:** 프론트

`/admin/reports`가 데모 데이터를 씁니다.

**이미 있는 것 (백엔드):**
- `GET /admin/reports?status=` — 상태별 신고 목록
- `PATCH /admin/reports/:id` — 상태 변경 (body: `{ status }`)
- 상태 값은 **`RECEIVED` | `IN_REVIEW` | `RESOLVED`** 셋뿐입니다 (다른 값은 400)
- `Report` 모델: `targetType`(ROOM/USER/POST 등), `targetId`, `reason`, `status`

**할 일:**
1. 클라이언트 함수 추가
2. 페이지 연결 + 상태 필터
3. `targetType`에 따라 신고 대상(숙소/유저/글)으로 가는 링크를 붙이면 좋습니다

**완료 기준:** 신고를 처리 완료로 바꾸면 목록에서 상태가 바뀜

### #4 내 리뷰 페이지

**난이도:** 중 · **영역:** 풀스택

`/me/reviews`가 데모 데이터를 씁니다. 게스트가 **자기가 쓴 리뷰**를 보는 화면입니다.

**문제:** `GET /reviews`는 `roomId`가 **필수**라, 작성자 기준으로 조회할 방법이 없습니다.

**할 일:**
1. **백엔드** — `GET /reviews/written`(가칭) 추가: `where: { authorId: req.user.id }`
   → `reviews.module.ts`의 `listForHost()`가 거의 같은 모양이니 그대로 본뜨세요.
   ⚠️ `:id` 라우트보다 **위에** 선언해야 합니다 (안 그러면 `written`을 id로 해석)
2. **프론트** — `lib/api/reviews.ts`에 클라이언트 추가, 페이지 연결
3. 리뷰에 호스트 답글이 달렸으면 함께 표시

**완료 기준:** 내가 쓴 리뷰만 보이고, 호스트 답글도 함께 뜸

### #12 Prisma 마이그레이션 도입 ⚠️

**난이도:** 중 · **영역:** 인프라 · **우선순위: 높음**

지금까지 `prisma db push`로만 스키마를 반영해왔습니다. `migrations/` 폴더가 없습니다.

**왜 문제인가:** 여러 명이 스키마를 바꾸기 시작하면
- 변경 이력이 없어 누가 뭘 바꿨는지 모름
- 롤백 불가
- 프로덕션 DB에 의도치 않은 변경이 나갈 수 있음

**할 일:**
1. 현재 스키마 기준으로 초기 마이그레이션 생성 (`prisma migrate dev --name init`)
2. 프로덕션에 `migrate resolve`로 베이스라인 설정
3. 이후 `db push` 금지, `migrate` 사용

**팀 합류 전에 하는 게 좋습니다.** 나중에 할수록 아픕니다.

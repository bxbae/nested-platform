# A. AI 엔지니어

> ⚠️ **현재 Nested에는 AI 기능이 전혀 없습니다.**
> 다른 역할과 달리 **기존 코드에 기댈 게 없습니다.** 설계부터 시작해야 합니다.

## A-1. 선행 작업 — 데이터 설계 (여기부터)

| 기능명 | 기능 설명 | 상태 | 우선순위 |
|---|---|---|---|
| **성향 데이터 스키마 설계** | 추천에 쓸 사용자 성향·선호 데이터 정의 | 🔴 | **1** |
| **성향 설문** | 가입 시 생활 패턴 조사 (흡연·소음·청결·생활시간대 등) | 🔴 | **1** |
| **성향 수정** | 가입 후에도 설문 내용 변경 | 🔴 | 2 |

> **데이터가 없으면 추천도 매칭도 불가능합니다.** 이게 가장 먼저입니다.
>
> **결정할 것:** `User`에 성향 필드를 추가할지, 별도 `UserPreference` 모델을 팔지.
> 후자를 권합니다 (설문 항목이 늘어날 수 있으므로).
>
> ⚠️ **스키마 변경은 E(총괄)와 반드시 협의.** 다른 역할도 스키마를 건드립니다.

## A-2. 룸메이트 매칭

| 기능명 | 기능 설명 | 상태 | 우선순위 |
|---|---|---|---|

## A-3. 숙소 추천

| 기능명 | 기능 설명 | 상태 | 우선순위 |
|---|---|---|---|
| **개인화 숙소 추천** | 성향·검색 이력 기반 | 🔴 | 3 |
| **추천 이유 표시** | 추천 근거를 자연어로 설명 | 🔴 | 3 |
| **유사 숙소 추천** | 상세 페이지의 "비슷한 숙소" | 🔴 | 3 |

## 기술 선택 (팀 논의)

- **규칙 기반 vs ML** — 초기엔 **규칙 기반(가중치 점수)으로도 충분**합니다.
  사용자 데이터가 없는 상태에서 ML은 학습할 게 없습니다. 데이터가 쌓이면 고도화하세요.
- **LLM 활용** — 추천 이유를 자연어로 생성하는 데 쓸 수 있습니다.
- **계산 시점** — 실시간 vs 배치 후 캐시. **Redis가 이미 있습니다.**

> ⚠️ **D(검색)와 협의:** 추천 결과를 검색 API에 녹일지, 별도 엔드포인트로 뺄지.

>   → `findOne`에서 `address`를 빼는 코드가 있습니다. **되돌리지 마세요.**

> ⚠️ **마이그레이션이 가장 급합니다.**
>
> **스키마를 건드리는 역할:**
> - **A** → 성향 필드 (`UserPreference` 신규) — **가장 큰 변경**
> - **B** → 프로필 (기존 `User.bio` 사용 → **변경 불필요**), 뱃지는 신규 필요
> - **C** → 공지/배너 모델 신규 (우선순위 낮음)
> - **D** → `Message.readBy`가 **이미 있어 변경 불필요**
>
> 즉 **A가 스키마 변경의 대부분을 차지합니다.**
> 그래도 마이그레이션 체계는 **A가 시작하기 전에** 갖춰야 합니다.
> **기능 개발 시작 전에** 정리하세요. → `MIGRATION_RUNBOOK.md`

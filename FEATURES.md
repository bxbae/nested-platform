# Nested — 기능 정의서 / 작업 배분

> **주의: Nested는 이미 상당 부분 구현돼 있습니다.**
> 아래 표의 **상태** 열을 반드시 확인하세요. ✅는 이미 동작하는 기능입니다.
> 다시 만들지 마세요.

**상태 표기**
- ✅ **완료** — 실제 API 연동 완료, 동작함
- 🟡 **연결 필요** — 백엔드는 있는데 화면이 데모 데이터를 씀 → **프론트만 작업**
- 🔴 **미구현** — 백엔드 없음 → **풀스택 작업**

## 7. 관리자

| 기능명 | 기능 설명 | 상태 | 우선순위 | 담당자 |
|---|---|---|---|---|
| 숙소 승인 | 등록 대기 매물 검토·승인 | ✅ | - | - |
| 숙소 거부 | 대기 매물 삭제 (예약 있으면 차단) | ✅ | - | - |
| **회원 관리** | 회원 목록·검색·정지 (`GET /admin/members` 있음) | 🟡 | **1** | |
| **신고 관리** | 신고 조회·상태 변경 (`GET /admin/reports` 있음) | 🟡 | **1** | |
| **관리자 대시보드** | 지표 카드 (`GET /admin/stats` 있음) | 🟡 | **1** | |
| **통계 / 매출 차트** | 월별 추이 (합계만 있음, 시계열 API 없음) | 🔴 | 2 | |
| **예약 관리** | 전체 예약 조회 | 🟡 | 2 | |
| **공지 / 배너** | 모델·API 모두 없음 (완전 신규) | 🔴 | 3 | |
| 에러 페이지 | 404·500 페이지 | 🟡 | 3 | |

> **🟡 우선순위 1 세 개(회원·신고·대시보드)가 신규 합류자 첫 작업으로 가장 좋습니다.**
> 백엔드가 이미 돌고 있어서 **프론트 클라이언트만 쓰면 끝**입니다.
> `coliving/src/lib/api/admin.ts`와 `app/admin/approvals/page.tsx`를 그대로 본뜨세요.
>
> **정확한 요청 형태:**
> - 회원 정지: `PATCH /admin/members/:id/suspend` — body `{ suspended: boolean }`
> - 신고 상태: `PATCH /admin/reports/:id` — body `{ status }`,
>   값은 **`RECEIVED` | `IN_REVIEW` | `RESOLVED`** 셋뿐 (다른 값은 400)

## 작업 배분 가이드

| 성향 | 추천 첫 작업 | 이유 |
|---|---|---|
| 프론트 위주 | 관리자 **회원 관리** | 백엔드 있음. 패턴 익히기 좋음 |
| 프론트 위주 | 관리자 **신고 관리** | 같은 패턴 + 상태 변경 |
| 풀스택 입문 | **내 리뷰** | 백엔드 라우트 1개 + 프론트 |
| 풀스택 | **프로필 수정** | `PATCH /auth/me` 신규 + 권한 처리 |
| 실시간 관심 | **실시간 메시지** | Socket.io 게이트웨이 이미 있음 |
| 인프라 | **Prisma 마이그레이션** | 가장 급함 |
| 큰 작업 | **호스트 대시보드** | 집계 API 설계부터 |

**충돌 방지:** 모듈 단위로 나누면 겹치지 않습니다.
`admin` / `host` / `me` / `messages` / `reservations` 중에서 고르세요.
